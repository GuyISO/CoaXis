# CoaXisViewer 修正ルール（コード・コメント）

## 1. 目的
この文書は、日常的な改修で品質を揃えるための最小ルールを定義する。

## 2. コード修正ルール

### 2.1 変更粒度
- 1コミットで扱う責務は1つに絞る（例: バグ修正のみ、コメント整形のみ）。
- 振る舞い変更とリファクタリングを同時に行わない。

### 2.2 Null安全
- AutoLoadシングルトン参照は利用前に初期化順序を意識する。
- イベント購読・解除時は対象Hubの存在を確認する。
- Node取得（GetNodeOrNull）結果がnullの場合は黙殺せずログを残す。

### 2.3 例外処理
- catch (Exception) の多用を避ける。
- 想定する例外型を明示し、ログに原因を残す。
- 回復不能例外は握り潰さない。

### 2.4 リソース管理
- StreamWriterなどの保持リソースは _ExitTree で解放する。
- モデルロード失敗時は生成ノードを後始末（QueueFree）する。

### 2.5 パフォーマンス
- 毎フレーム処理での不要アロケーションを避ける。
- 再帰探索処理は深さ・循環参照リスクを考慮する。

### 2.6 イベント設計
- EventHub経由で状態通知するときは初期状態通知も忘れない。
- Request（要求）と Notify（通知）の責務を混在させない。

## 3. コメント修正ルール

### 3.1 XMLコメント整形
- summary, param, returns, remarks は改行して独立させる。
- タグ閉じ忘れを禁止する。
- 1行に複数タグを詰め込まない。

### 3.2 曖昧語の禁止
- XXX, TODOのみ、仮名表現を公開コメントに残さない。
- 操作モード名やイベント名は実コードの識別子に揃える。

### 3.3 コメント品質
- 「何をするか」と「なぜ必要か」を短く書く。
- コードを読めば分かる逐語説明は避ける。

### 3.4 新規サービス追加時の必須コメント
- AutoLoad / Service を新規追加するときは、クラス summary に「責務」「読込元」「フォールバック方針」を明記する。
- 外部ファイル読込処理は、探索順（優先順位）を comments または remarks で残す。
- public メソッドには XML コメント（summary / returns / param）を付与する。
- 既定値へフォールバックする処理には、理由が分かる1行コメントを置く。

## 4. C#ファイル構成ルール
ユーザー操作イベントを含むクラスは region の順序を以下に揃える。

1. Lifecycle
2. Events
3. Public Methods
4. Internal Helpers

## 5. PRセルフチェック
- 仕様どおりに動くか
- 新規Null参照を導入していないか
- イベント購読解除漏れがないか
- ログやエラー原因が追跡可能か
- XMLコメントの崩れがないか
